# 通用工程基础与技术选型

## 文档定位

本篇先定义所有 AI Coding 项目的通用基础，再补充 Agent 项目和前端项目的选型建议。

## 一、通用基础层

以下内容默认适用于所有项目。

### 运行环境

- 运行时优先使用稳定 LTS 版本，Node.js 建议 `20+`，最低不低于 `18`
- 同一仓库只允许一种包管理方式，禁止混用 `npm`、`yarn`、`pnpm`
- 同一仓库的构建、测试、脚本命令应保持统一，不要同类能力出现多套入口

### 语言与类型

- 优先使用 TypeScript；若因历史原因使用 JavaScript，需控制模块规模并逐步补齐类型边界
- 类型系统应承担输入、输出、配置、内部模型的约束职责
- 常量优先使用 `as const` 与联合字面量类型，避免 `enum` 带来的额外心智负担

### 配置与校验

- 配置文件只保存非敏感默认项
- 身份、token、密钥、环境差异项通过环境变量或本地私有配置注入
- 输入参数、配置对象、结构化输出推荐使用 `zod` 或 JSON Schema 做校验

### 工具与自动化

- 必须有统一的格式化、类型检查、测试和 CI 校验入口
- 规范优先依赖自动化卡口落地，而不是靠人工记忆执行

## 二、Agent 项目补充

以下内容适用于 Skill、自动化脚本、工作流编排、工具调用型项目。

### 推荐选型方向

- Prompt、规则、模板优先文件化管理，不内嵌在业务逻辑长字符串里
- 外部能力通过 `tools`、`adapters`、`services` 等层统一封装
- 输出优先结构化，避免把纯自然语言长文本当作系统间接口
- 需要保留 fixtures、eval cases、回放数据，便于回归验证

### 不默认强制的内容

以下内容不应作为纯 Agent 项目的默认约束：

- React
- React Router
- Ant Design 等 UI 组件库
- Tailwind / CSS Modules 等样式体系
- 页面路由与组件分层

只有当 Agent 项目包含控制台、页面或可视化界面时，才引入这些前端能力。

## 三、前端附录

只有项目包含页面和 UI 层时，才启用以下选型建议。

### 推荐技术栈

- 框架：`React 18+`
- 语言：`TypeScript 5.x`，并开启 `strict`
- 路由：`React Router v6`
- 样式：`CSS Modules + Tailwind`
- 请求层：推荐 `Axios + React Query`
- 组件库：同一项目只保留一套主组件体系

### 选型原则

- 同一层能力只保留一种主方案，避免 AI 和人工协作时上下文混乱
- 优先选择类型友好、文档完善、社区成熟、AI 训练覆盖充分的方案
- 如果仓库已有既定规范，应服从仓库级规则，不再额外并行引入替代方案

## 四、统一原则

1. 通用基础层永远优先于专项约束。
2. Agent 项目默认不强绑前端技术栈。
3. 前端项目如果接入 Agent 能力，应同时遵守 Agent 项目补充章节。
4. 所有技术选型最终都应写入项目上下文文件，避免靠聊天重复传递。
