# 通用编码与类型规范

## 文档定位

本篇定义所有 AI Coding 项目都应遵守的编码与类型底线，并补充 Agent 项目和前端项目的专项约束。

## 一、通用编码规范

### 代码职责

- 每个模块优先只做一件事，避免把输入处理、业务逻辑、外部调用、输出格式化全部揉在一起
- 公共逻辑优先抽到独立模块，避免复制粘贴
- 复杂分支要显式表达边界条件、空值处理和异常路径

### 类型规范

- 类型必须承担边界约束职责，禁止把大量不确定性推给运行时碰运气
- 禁止滥用 `any`
- 仅在确有必要时使用类型断言 `as`，且需要说明为什么无法在类型层表达
- 回调、配置、数据模型必须使用精确类型，不用 `Function` 或 `(...args: any[]) => void`

### 命名与表达

- 命名优先表达真实业务语义，不要使用含义模糊的占位名
- 优先通过结构分层提升可读性，而不是堆注释解释糟糕结构
- 注释重点说明“为什么这样做”，而不是重复代码表面行为

## 二、Agent 项目补充

### 编排与工具边界

- 主流程只做 orchestration，不直接散落 shell 命令、HTTP 请求或 SDK 细节
- 外部系统调用统一通过 `tools` / `adapters` 层封装
- Prompt 组装、数据解析、结果格式化应拆成独立模块，而不是混在一个主函数里

### 输出与失败处理

- 能结构化输出时，优先输出对象或 JSON，而不是依赖自由文本格式
- 输出字段必须稳定命名，避免下游依赖脆弱字符串匹配
- 对超时、空返回、重试失败、部分失败等情况要有显式处理，而不是直接吞错

### AI 审查重点

审查 Agent 相关代码时，优先看：

1. 是否把高风险调用直接写死在主流程里
2. 是否缺少 timeout、重试边界、失败兜底
3. 是否把 Prompt、规则、解析逻辑写成不可维护的大段字符串
4. 是否缺少结构化输出约束和 schema 校验
5. 是否把敏感配置直接写入代码或默认配置文件

## 三、前端附录

### 前端编码补充

- 全部使用函数式组件与 Hooks，避免引入额外的组件范式混乱
- 页面装配、展示逻辑、状态逻辑、请求逻辑应分层
- 受控组件统一使用 `value / onChange`
- UI 相关异步请求应收敛到统一数据层处理，不在组件内部随意分散实现

### 前端评审重点

审查前端代码时，优先看：

1. 是否破坏既有目录与命名约定
2. 是否引入了规范外依赖或重复能力依赖
3. 是否把页面逻辑、样式、请求和状态硬耦合在一个文件里
4. 是否出现过度组件抽象或伪通用组件

## 四、统一原则

1. 通用编码规范优先于专项写法偏好。
2. Agent 项目优先保证稳定编排与结构化输出。
3. 前端项目优先保证交互一致性与组件边界清晰。
4. AI 生成的大段代码必须人工复核边界条件和异常路径。
