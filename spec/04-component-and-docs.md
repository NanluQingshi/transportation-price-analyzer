# 前端附录：组件、交互与文档规范

## 文档定位

本篇是前端附录，只在项目包含页面、组件和交互界面时生效。纯 Agent / Skill 项目可以跳过本篇。

## 一、组件建设原则

- 不追求大而全的公共组件库，只沉淀高频、复杂、领域相关、难以一次性正确生成的组件
- 基础 UI 能力优先使用统一组件体系，不做无意义二次封装
- 业务组件优先在业务边界内维护，避免过早抽象成伪通用组件

## 二、组件分层建议

- 基础组件：按钮、输入、布局、反馈等通用能力
- 领域基础组件：金额、审批流、身份选择、复杂选择器等业务共性能力
- 页面私有组件：只服务单页面或单流程的局部交互，不进入公共层

## 三、Props 与类型规范

- Props interface 独立导出，命名为 `ComponentNameProps`
- Props interface 与组件实现放在同文件顶部，降低检索成本
- 布尔 prop 使用语义清晰的命名，默认值优先保持可推断
- 两态用 boolean，三态及以上优先使用联合字面量类型
- 回调参数、泛型边界、可选值语义都要给出精确类型

## 四、文档与注释规范

- 每个组件顶部写一句话，说明这个组件解决什么问题
- 每个组件提供最小可运行示例，便于 AI 和开发者快速理解
- 每个 prop 写单行 JSDoc，说明用途而不是重复类型
- 默认值使用 `@default` 标注，典型用法使用 `@example` 标注

## 五、交互与维护建议

- 受控模式优先统一为 `value / onChange`
- 组件抽取前先验证是否真的会被多个场景复用
- 抽象失败时，优先回退到业务内实现，避免形成长期维护负担
- 对外 API 一旦被多个页面依赖，应优先保持稳定，减少破坏式修改

## 六、推荐目录

```text
src/components/AmountInput/
├── index.tsx
├── index.test.tsx
└── index.module.css
```

## 七、适用边界

以下场景通常不需要启用本篇：

- 纯 Skill 项目
- 纯 CLI 项目
- 纯自动化工作流项目
- 没有页面层的 Agent 项目
