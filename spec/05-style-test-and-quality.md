# 通用质量卡口、Agent 补充与前端附录

## 文档定位

本篇先定义所有 AI Coding 项目的通用质量卡口，再补充 Agent 项目和前端项目的专项要求。

## 一、通用质量卡口

### 开发时

- 使用统一的格式化与静态检查工具，保证基础风格与低级错误能被即时发现
- 保存自动修复可以开启，但不能替代人工理解代码结构

### Commit 时

- 提交前至少执行格式化、类型检查和关键测试
- 禁止通过跳过本地校验来规避规范

### PR 时

- CI 必须检查类型、测试和静态规则，任一失败都阻断合入
- 单次改动建议保持在可 review 范围内，过大改动应主动拆分

### 发布前

- 对关键入口、核心输出、回归风险点进行最小集验证
- 有外部依赖的项目应明确上线前验证清单，而不是只看构建成功

## 二、Agent 项目质量卡口

### 配置与安全

- 启动前必须校验配置文件、环境变量、身份字段是否完整
- 敏感信息不得进入默认配置、测试样例和日志正文

### 输出与回归

- 结构化结果必须经过 schema 校验
- Prompt、解析逻辑、输出模板变更后应支持回放测试或评测
- 对关键任务保留 fixtures，确保能做低成本回归

### 运行稳定性

- 外部调用必须设置 timeout、重试上限和失败兜底
- 发送消息、写文件、调用审批、触发执行等副作用动作应支持 dry-run、mock 或安全开关
- 定时任务和批量任务要支持幂等重跑，避免重复执行带来副作用

### 观测性

- 日志至少要能定位任务时间、步骤、工具、错误位置
- 日志和报告必须脱敏，不能泄露凭证、身份信息和敏感链接参数

## 三、前端附录

### 样式规范

- 只在项目包含页面和 UI 层时启用样式规则
- 样式优先遵守统一主题、统一变量和统一命名方式
- 组件样式要兼顾一致性与可维护性，避免局部魔改和长期堆叠

### 测试规范

- 工具函数、核心状态逻辑、复杂交互必须具备测试覆盖
- UI 测试优先覆盖关键交互路径，而不是堆快照
- 纯展示组件可根据复杂度决定是否测试，不机械追求覆盖率

## 四、统一原则

1. 所有项目都先满足通用质量卡口。
2. Agent 项目额外关注配置、输出、回放、超时、幂等与观测性。
3. 前端项目额外关注样式一致性、交互稳定性与 UI 测试。
4. 自动化卡口是底线，不替代人工 review 和业务理解。
