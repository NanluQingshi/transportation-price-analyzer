# 通用 AI 协作规范与专项补充

## 文档定位

本篇先定义所有 AI Coding 项目通用的协作方式，再补充 Agent 项目和前端项目的专用要求。

## 一、通用 AI 协作规范

### 项目上下文文件

每个项目根目录都应维护上下文文件，例如 `AGENTS.md`、`CLAUDE.md` 或同类文件，至少包含：

- 技术栈与版本
- 目录结构约定
- 编码与评审规则
- 关键禁用项
- 任务边界与输出要求

原则：规范优先写进文件，不反复依赖聊天上下文传递。

### 任务描述规范

向 AI 提需求时，尽量明确以下信息：

- 修改位置：给出文件路径或目录范围
- 目标对象：给出模块名、页面名、流程名、接口名或工具名
- 边界限制：明确“只做什么，不做什么”
- 预期输出：说明是改现有文件、补测试、补文档，还是新增能力

### 任务粒度控制

- 单次任务尽量控制在一个功能点内
- 大改动应拆成多个可验证的小任务
- 不要把“优化整个模块”这类大而虚的目标直接丢给 AI

### 通用评审重点

人工评审 AI 生成内容时，优先看：

1. 是否偏离任务边界
2. 是否引入了不必要的新方案或新依赖
3. 是否缺少边界处理、异常分支和空值处理
4. 是否破坏既有目录、命名和规范约定

## 二、Agent 项目专项补充

### Prompt 与规则管理

- Prompt、规则、few-shot、输出模板应独立存放并可版本管理
- Prompt 变更尽量附带回放样例或评测样例
- 不要把大段 Prompt 直接写死在业务主流程里

### Tool 边界

- 主流程只做编排，不直接散落外部调用实现细节
- 所有工具调用应统一封装，并明确输入、输出、失败策略
- 高风险动作必须区分“可自动执行”和“需要人工确认”

### 输出与运行约束

- 优先输出结构化结果，而不是依赖自然语言长文本作为系统接口
- 关键任务需明确输入、输出、失败策略、超时策略和回滚策略
- 对批量任务、定时任务和副作用任务，优先要求幂等设计

## 三、前端附录

### 前端任务补充

如果任务涉及页面或交互，建议在描述中额外说明：

- 所在页面或路由
- 涉及的组件边界
- 数据来源与请求范围
- 交互行为与状态变化
- 是否包含样式、测试和文档更新

### 前端评审补充

审查前端 AI 产出时，额外关注：

- 页面装配是否清晰
- 组件抽象是否合理
- UI 变更是否影响既有交互一致性
- 是否遗漏关键交互测试或文档示例

## 四、统一原则

1. 通用协作规范对所有项目生效。
2. Agent 项目重点关注 Prompt、Tool、输出和运行边界。
3. 前端项目重点关注页面边界、组件抽象和交互一致性。
4. AI 负责提效，但最终质量由规范文件、自动化卡口和人工评审共同保证。
